**Hackathon Judgement Criteria**
--------------------------------

Bare Minimum Requirements
-------------------------

Hackathon submissions **MUST** follow these rules to reach the entry requirements of the hackathon. Even if your project is high-quality, failure to meet these requirements will result in disqualification.

**These requirements apply across all tracks of the hackathon.**

1.   All projects must be open-source.
2.   These archived codebases are strictly for the purpose of common good projects and will not be used for commercial purposes.
3.   Only the submitted codebase at the final judgment deadline will be archived.
4.   The hackathon encourages building on top of existing open-source code. However, projects that fork from established open-source repositories with more than **70%** codebase similarity to the original will be immediately disqualified.
5.   All project team members must verify their identity through the Polkadot Official Discord channel.
6.   A valid commit history is required to showcase your team’s contributions during the hackathon.
7.   The commit log must clearly reflect active contributions during the event timeline.
8.   Only the code contributed during the hackathon will be considered for scoring.

Project Winner Criteria
-----------------------

To qualify as a winning project, the following deliverables and requirements must be met:

Polkadot On-Chain Identity:
---------------------------

The winning team must set up a Polkadot wallet with an on-chain identity enabled for identity verification.

**Guide**: How to set up your on-chain identity on**Polka-Assembly**[https://openguild.wtf/blog/polkadot/polkadot-opengov-introduction](https://openguild.wtf/blog/polkadot/polkadot-opengov-introduction)

Quality Documentation:
----------------------

Submissions must include a well-structured project description. Acceptable formats include:

*   GitHub README file
*   Notion page
*   Website or GitBook
*   Docx / PDF file

UI/UX Experience:
-----------------

A good user interface and smooth user experience will be taken into account alongside the technical functionalities.

Demonstration:
--------------

Every submission must include a demo video or screenshots. The judges should be able to test your features with either:

A hosted deployment
-------------------

A local installation guide in the README

Vision & Commitment:
--------------------

Projects that show a clear roadmap, innovative idea, and proof of future commitment will receive extra points during evaluation.

Track Relevance:
----------------

The solution must be valid for the track you are applying to (EVM, PolkaVM, Cross-chain, Tooling, etc.).
